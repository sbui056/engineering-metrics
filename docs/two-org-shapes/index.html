One engine · two repositories

Two org shapes, one engine

I built a tool that ranks open-source contributors by how much the codebase depends on them, not by how much they type. Pointing it at two repositories told two very different stories.

The contributor my engine ranks first on FastVideo has 54 commits. The contributor with 352 commits, who has also reviewed more of his colleagues' work than anyone else there, comes in second. On ComfyUI the picture is stranger: one person wrote 59% of all the code that survives in the current codebase, and commit count turns out to be a weak guide to who the project depends on. The impact ranking and the commit-count ranking agree far less there (a rank correlation of 0.46) than they do on FastVideo (0.75). Both readings came out of the same pipeline with the same thresholds and the same weights, and nothing was retuned between the two runs.

That contrast taught me more about the method than either leaderboard alone did. If the same formula had told the same story about a small academic lab's project and a 120,000-star community tool, I would suspect it was measuring a property of the formula rather than a property of the projects.

What "impact" means here

The engine defines impact as the degree to which an engineer's work is depended on, trusted, and hard to replace, rather than the amount of work there is. One term needs a plain definition up front: "surviving code" means the lines still present in the current version of the codebase, attributed to their authors by git blame. Code that was written and later rewritten by someone else does not survive, and that distinction does a lot of work in what follows.

Four signals feed the score, and each one answers a different form of the question "would this person be hard to replace?"

  • Ownership concentration is the share of surviving code an engineer is the leading author of, weighted by file size. If they left tomorrow, this is the code that has their name on the blame.
  • Code survival measures how much of what they wrote is still there, adjusted for how long they have been contributing. Code that lasts is code other people build on.
  • Coupling criticality asks how central their files are in the graph of files that change together, using PageRank. Owning a file that everything else moves with is different from owning a leaf.
  • Review leverage counts reviews given, weighted by how many distinct people they reviewed for. This is trust made visible: whose judgment does the project route its changes through?

0.25 × 4

the score is the average of four percentiles; the weighting is disclosed as a choice rather than presented as tuned.

Each signal is converted to a percentile and the four percentiles are averaged with equal weights. Both choices are deliberate. Percentiles, because repository activity is heavily skewed and a z-score would let one prolific outlier compress everyone else's signal toward zero. Equal weights, because there is no ground truth to calibrate against, and a set of tuned-looking weights would imply a precision that does not exist. Unit-weighted composites are also famously hard to beat in exactly this kind of setting (Dawes, 1979, "The robust beauty of improper linear models"). The full mechanics, including identity resolution, co-author crediting, and rename tracking, are in the methodology.

provenance

every number in this essay was generated by the build script from the dashboards' data payloads; none was typed by hand.

The pipeline itself is plain: git log, git blame with whitespace and copy detection, PageRank on a co-change graph, and the GitHub reviews API. There is no AST parsing, no per-language tooling, and no language model judging anyone's diffs. Every number on the dashboards is computed from the repository at build time, which will matter later in this story.

The baselines I rejected, measured

Commit count and lines of code were rejected as headline signals before any code was written, because they measure typing rather than dependence. Having built the alternative, I can now say precisely what rejecting them bought, because the disagreements land exactly where a naive ranking does damage.

Commit count cannot see rewrites. Zhang Peiyuan is second on FastVideo by commit count, with 132. His ownership and coupling percentiles are top-five material, but his survival percentile is 0.26, because much of what he wrote early has since been rewritten, and he lands at impact #9. Honesty requires the caveat that cuts the other way: blame-based survival structurally undercounts foundational work that others later refactored, so part of that 0.26 is real churn and part is a known bias of the method. The dashboard presents him with the evidence attached, not as "worse than #8."

Reviews are invisible in a diff stat. Wenxuan Tan reaches impact #10 largely on review leverage, with 127 reviews across 18 distinct authors. That is trust the repository visibly runs on, and no amount of staring at his line counts would surface it.

Quiet ownership beats loud activity. Shao Duan has 16 commits and has never filed a review, and the engine still ranks him #11, because he is the leading author on 138 files and has one of the highest survival rates on the board. An eleven-commit contributor sits at #4 for a similar reason: the small subsystem he owns is central and his code has survived. At the other end of the scale, FastVideo has 41 contributors with exactly one commit. Commit count ties all 41 of them at rank 47. The impact score spreads them from #24 to #87, depending on whether that one commit survived and how load-bearing the code it touched is. A ranking that cannot distinguish a surviving core-pipeline patch from a reverted typo fix is not measuring anything useful.

The #1 spot itself makes the cleanest case. alexzms is fourth by commit count and first by impact: what he owns is central, and what he wrote survived, so the score does not care that there is less of it. William Lin, the second-ranked contributor, is no less remarkable. He is the leading author on 627 files and has given 517 reviews across 55 people, and he holds the top percentile on three of the four signals. The fourth signal, survival, is where high volume shows its cost, because writing that much code means some of it gets rewritten, including by yourself.

Two org shapes

Running the engine on a second repository started as a portability test and ended as the finding.

FastVideo is a distributed lab. Four people hold half the surviving code. The most prolific contributor is the leading author on 322 files that nobody else majors in, though all but 132 of those files have at least one other contributor, and Shao Duan is already the strongest second owner on 30 of them. The top ten includes people whose main contribution is review trust rather than line ownership. Commit count correlates with impact at 0.75 here, so activity and importance mostly travel together, and the interesting stories are the exceptions.

ComfyUI is a solo cliff. One person, the founder comfyanonymous, holds 59% of all surviving code. The Gini coefficient of code ownership is 0.99, which is close to the maximum the statistic can express. He is the sole major owner of 191 files that carry roughly 20% of the co-change graph's total centrality, and 102 of those files have never been touched by anyone else. On this repository the departure question stops being a thought experiment and becomes the org chart. This is also the repository where commit count is least informative, with a rank correlation of 0.46, because the community around that core contributes in shapes that activity metrics flatten: the #2 contributor has 41 commits to the founder's 2,933, and earns the rank through owned subsystems and one of the largest review footprints in the project.

Neither shape is a verdict. Concentration reads two ways, as mastery and as fragility, so the engine scores it positively while flagging every single-owner file as bus-factor risk on the same screen. FastVideo's shape suggests resilience and succession candidates. ComfyUI's shape shows extraordinary individual leverage next to a succession-planning problem. Both readings appear on the dashboards side by side, because collapsing them into one number would be an editorial opinion pretending to be math.

The machinery has to be honest

The design decision I would defend hardest has nothing to do with any single signal. It is the rule for what happens when the data is incomplete.

Review data comes from the GitHub API, and on a repository with 4,511 pull requests a fetch can be cut short by rate limits or a cold cache. When that happens, the engine neither guesses nor punishes. Contributors whose review activity is unknown are scored at the median for that signal, and the page badges them as estimated, so a gap in data availability cannot masquerade as "worst engineer" and cannot silently inflate anyone either.

#6 → #19

ComfyUI produced a natural experiment. One contributor there has exactly one commit: a test-infrastructure drop whose lines survived for years as scaffolding that other people's tests are built on. While review data was partial, the median imputation placed them in the top ten of 311 contributors, with the badge showing. When the fetch completed and their review count was confirmed to be zero, they settled at #19 (they sit at #21 today, in a population that has since grown to 313). That is still the top 6%, on one commit, because the method genuinely values surviving, load-bearing code. The point is that the difference between "top ten, partially estimated" and "#19, fully measured" was visible the whole time. Better data moves ranks, and nothing gets smoothed away.

The same instinct shows up in smaller ways. The leaderboard is grouped into tiers whose width is derived from the score distribution, because a gap of 0.0005 between two scores at this sample size is noise, and presenting it as a real ranking difference would be false precision. FastVideo's 87 contributors fall into 58 tiers rather than 87 supposedly distinct ranks. The correlation between commits and impact that each dashboard displays is computed from that page's own data at build time rather than quoted from anywhere else. Even the prose adapts: a sentence about the zero-review block only renders if it is true for that repository.

What it can't see

Every signal is a proxy, and the failure modes are documented on the dashboards rather than waved away. Blame only attributes lines that survived, so foundations that were later refactored are undercounted. Coupling cannot see a load-bearing file that rarely changes, and high centrality sometimes measures entanglement, which is a design smell rather than a virtue. Review leverage measures reach rather than depth, so a one-line approval counts like a substantive review. Nothing here sees mentoring, design work, incident response, or unblocking a teammate at 2 a.m.

All of it can also be gamed. If this formula were published as a target, Goodhart's law would take over: territorial code hoarding pumps ownership, split PRs pump activity, and rubber stamps pump review reach. It works as an instrument for understanding how a codebase depends on its people. It would fail as a compensation formula, and I would argue nothing in this category should be used as one.

That is also why the strongest evidence for the method is the contrast this essay opened with. The engine was not tuned to make FastVideo look collaborative or ComfyUI look concentrated. The same thresholds, windows, and weights produced both readings, and both match what people who know these projects already believe about them.

Both dashboards are live, refresh themselves weekly, and show every caveat above on the page: FastVideo and ComfyUI. The engine is open. Point it at your own repository and see what shape you are in.
